# MuseForge Product Specification

## 1. Product definition

MuseForge is a local-first AI image production workspace for e-commerce visual, operations, and review teams. It turns folders, reference images, prompt documents, generation scripts, and manual review decisions into one observable production system.

The product is deliberately not centered on a single free-form prompt box. Its primary object is a traceable work item with:

- a product identity and evidence-backed facts;
- a standalone or accessory-combination task;
- a required e-commerce shot type;
- curated references and a reference manifest;
- structured prompt content and generation parameters;
- one or more temporary candidates;
- a reviewer decision and formal delivery location.

## 2. Target users

### E-commerce visual operator

Prepares products, selects tasks and shot types, launches local batches, and monitors generation progress.

### Designer

Uses selected candidates as scene backgrounds or layers, adjusts composition and copy, and exports delivery-ready images.

### Reviewer or merchandiser

Compares candidates, validates product fidelity and commercial usability, keeps approved images, and removes rejected images.

### Local workspace administrator

Configures image providers, workspace paths, live-generation gates, and local execution limits.

## 3. Core workflow

```text
Ingest source material
  → establish product truth
  → prepare five prompt types
  → curate references
  → select task × shot cells
  → run local generation
  → review temporary candidates
  → promote selected images
  → refine on canvas
  → export or deliver
```

The workflow has two complementary paths:

1. **E-commerce batch production** — the primary workflow, optimized for repeatable product-image output.
2. **Canvas refinement** — a single-image workspace for composition, text, layer, and delivery adjustments.

## 4. Information architecture

| Area | User purpose |
| --- | --- |
| Project overview | See output volume, review demand, active pipeline, and recent work |
| Creation canvas | Refine one product/task/shot document and its structured prompt |
| Task matrix | Select products, tasks, shot types, candidate counts, and local concurrency |
| Assets and facts | Inspect source files, product truth, curated references, and manifests |
| Generation queue | Observe real server-side runs, progress, failures, and review readiness |
| Review and delivery | Compare candidates, keep selected images, clean rejected images, and enter canvas |
| Connections and settings | Inspect local services, providers, paths, and generation safety state |

## 5. E-commerce task model

### Task axis

- `单品`: the standalone main product.
- `<accessory name>`: the main product combined with one prepared accessory.
- prepared variants may be represented as separate task folders when explicitly generated by the Skill.

### Shot axis

| ID | Display name | Primary purpose |
| --- | --- | --- |
| `main` | 主图 | Immediate product recognition and clean hierarchy |
| `size` | 尺寸图 | Evidence-backed dimensions and scale communication |
| `lifestyle-scene` | 场景图 | Realistic usage context and physical interaction |
| `detail` | 细节图 | Material, construction, texture, and functional detail |
| `comparison` | 对比图 | Clear differentiation without unsupported claims |

Each selected `task × shot` cell produces an independent review group. Candidate count is configurable from 1 to 6.

## 6. Readiness and lifecycle states

Three concepts remain logically separate even when the UI presents a concise combined label.

### Readiness

- `draft`: preparation is incomplete.
- `blocked`: a hard gate such as a prompt or reference manifest is missing.
- `ready`: the work item can be submitted.
- `stale`: source truth, references, prompts, or rules changed after an earlier output.

### Execution

- `queued`
- `running`
- `completed`
- `failed`
- `cancelled` as a future worker-control state

### Review and storage

- `pending / staged`: temporary local candidate, not a formal asset.
- `selected / promoted`: reviewer-approved file moved into the formal task output directory.
- deleted: candidate row and file removed after explicit review action.

## 7. Generation gates

Live generation must not start unless all applicable gates pass:

- the product exists as one safe workspace folder;
- at least one explicit task and shot are selected;
- structured prompts exist for the selected work items;
- `参考图/` and `reference_manifest.json` agree exactly;
- standalone reference counts and roles are valid;
- combination reference counts and roles are valid;
- visible text and sensitive claims comply with the product-truth rules;
- the image provider is configured;
- `MUSEFORGE_ENABLE_LIVE_GENERATION=true` is explicitly set server-side;
- requested candidate count and concurrency remain within configured limits.

Blocked matrix cells cannot be submitted as part of a batch.

## 8. Review semantics

Generated images enter a temporary run directory first. Review decisions change storage state:

```text
generated candidate
  ├── keep → atomic promotion → formal task asset
  └── delete → remove temporary file + candidate record
```

Important interaction rules:

- selecting a checkbox does not move a file until the user confirms “keep selected”;
- bulk cleanup excludes already promoted candidates;
- sending a staged candidate to canvas first promotes it;
- a promoted candidate uses a stable workspace asset URL in saved canvas documents;
- deleting a promoted file requires a separate warning because existing canvases may reference it.

## 9. Canvas behavior

Canvas documents are isolated by `product + task + shot` so different product workflows cannot overwrite each other.

Current interaction model:

- image and text layers;
- drag, resize, rotate, duplicate, delete, align, and layer ordering;
- select and hand tools;
- wheel zoom, fit view, and 100% view;
- Space for temporary hand mode;
- arrow-key nudging with Shift acceleration;
- locked background layer;
- structured prompt editing;
- undo and redo history for node edits;
- 1024 × 1024 PNG export without editing overlays.

Persistence requirements:

- loading must finish before editing begins;
- load failures must never overwrite a server document with defaults;
- saves for one canvas must be serialized;
- edits made while a save is running must produce a trailing save;
- route changes must write a synchronous session cache and attempt persistence;
- offline changes must retain `pendingSync` until a future online save succeeds;
- restored custom viewport state must not be replaced by an automatic fit operation.

## 10. Demonstration and live-data rules

Demonstration data must always be labeled and must not:

- call a real image provider;
- create or delete candidate files;
- be presented as server-side run progress;
- write a reviewer decision to production metadata.

When the API is available, queue, workspace, and review pages use real data even when live generation remains disabled.

## 11. Success criteria

The current local-first release is successful when a user can:

1. place product and accessory source material into the workspace contract;
2. prepare prompts and curate required references;
3. submit valid task/shot cells as one batch;
4. observe a non-blocking server-side run;
5. review generated candidates grouped by work item;
6. promote only selected images and delete rejected images;
7. send a promoted image into its correct canvas;
8. return to any product/task/shot canvas without data crossing contexts;
9. export a clean 1024 × 1024 image.

## 12. Roadmap

### Near term

- manifest editing and publishing through the API;
- run cancellation and retry controls;
- persisted event timeline and worker diagnostics in the queue UI;
- real selected-output history in the canvas asset panel;
- multi-select, snapping, distribution, crop, mask, and visibility controls;
- marketplace and channel-specific artboard templates.

### Deployment evolution

- persistent worker processes with heartbeats and leases;
- local companion for cloud-hosted review UI;
- temporary preview storage with TTL where cross-device review is required;
- permanent object storage only for retained outputs;
- authentication, organization workspaces, permissions, and audit policies;
- remote GPU and multi-provider routing.
