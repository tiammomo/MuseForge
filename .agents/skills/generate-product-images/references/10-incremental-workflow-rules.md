# 增量工作流与 Token 控制规则

## 目标

减少 Codex 对未变化资料的重复读取、视觉理解和文字复述，同时保持首次处理、变更处理和生图前验证的质量标准不变。不得通过减少参考图检查、删除关键约束、弱化物理逻辑或缩短必要的成图提示词来节省 token。

## 两类 Token 分开管理

- Codex token：用于理解商品资料、视觉筛图、修正事实、审查提示词和对话输出。
- Image2 输入 token：每张图都必须携带足够的独立约束；不得假设并发请求共享上下文。

优先降低 Codex 的重复理解成本。Image2 prompt 只删除语义重复，不删除商品身份、数量、颜色、尺寸依据、物理交互、主次层级、场景、文字语言和负面约束。

## 一次深审，形成三份记录

首次处理商品后，在 `组合/<商品>/_workflow/` 保存紧凑的机器可读记录：

1. `source_fingerprint.json`：列出参与判断的源文件相对路径、字节数、修改时间和 SHA-256。
2. `product_profile.json`：只保存经过证据确认的商品事实、禁止推断项、SKU/数量差异、场景域、交互机制、尺寸性质和美国站文案限制；每条关键事实附来源路径。
3. `quality_record.json`：记录已视觉检查的图片、接受/排除理由、适用角色、场景与物理逻辑结论、已审查任务及规则版本指纹。

记录是推理缓存，不是新的事实来源。发生冲突时必须回查原始资料。

## 缓存命中条件

同时满足以下条件才可复用：

- 所有相关源文件的 SHA-256 与记录一致。
- 当前 SKU、任务名称和配件身份一致。
- 规则文件指纹未影响该任务。
- 精选参考图与 `reference_manifest.json` 一致。
- 上次质量记录没有未解决问题。

任何相关图片、说明文档、规格、规则或配件资料发生变化，只失效受影响的事实和任务，不连带重做无关组合。

## 参考图复用

- 主商品候选图对同一 SKU 只做一次完整视觉审查，结论写入质量记录。
- 配件候选图对同一配件只做一次完整视觉审查。
- 每个任务仍必须拥有自己的 `参考图` 和 `reference_manifest.json`，但可从已批准集合复制，不重复进行相同视觉推理。
- 组合可根据场景和交互需要从批准集合中选不同子集；不得机械地让所有组合使用完全相同的五张图。
- 新场景需要批准集合未覆盖的安装、尺寸或材质证据时，必须补做视觉审查。

## 提示词生成与审查

- 用确定性脚本生成五类提示词基线，不让 Codex 从空白重复撰写成套 JSON。
- 结构校验始终全量执行：数量、字段、长度、文件名、英语规则、参考图边界、组合层级和交互字段均不可抽样。
- 语义复审采用变更驱动：未变化且已有合格记录的 prompt 不重复全文读取；发生变化的任务完整复审五条。
- 场景图或交互机制发生变化时，必须完整复审环境、承载面、接触点、受力方向、完成状态和禁止状态。
- 尺寸来源、SKU 数量或组合身份发生变化时，必须完整复审尺寸图和主图。

## 对话输出

- 不在聊天中粘贴完整 `prompts.json`、长规则或源文档。
- 只报告计数、路径、分类决策、缓存命中/失效原因、发现的问题和人工修正。
- 用户要求查看具体 prompt 时，只展示指定任务或指定图型。

## 推荐执行粒度

- 新商品：完整读取、完整视觉审查、完整五图语义审查。
- 已有商品新增一个配件：复用商品档案，只深审新配件及其五图组合。
- 只修改一个配件：只失效包含该配件的组合。
- 只补参考图：只重审受影响任务的参考边界和相关图型。
- 只重跑缺图：不刷新 prompt，不重读源资料，直接 `preview` 后由用户运行 `generate`。
